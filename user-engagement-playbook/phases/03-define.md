# Phase 3 – Define

Discovery gave you ground truth; this phase converts it into prioritized, framed, elaborated scope that a team can build. Three events, in strict order, because each consumes the previous one's output.

---

## Scope & Prioritization Workshop

**Purpose:** Force-rank the problems worth solving now, with stakeholder-endorsed priority as the output.

The operative word is *force*. Every stakeholder's problem is priority one until the room is required to produce a single ordered list. This workshop exists to make trade-offs explicit and endorsed, so that when capacity runs out (it always runs out), the cut line was already agreed to.

**Participants:** Business owners, stakeholder representatives, program leadership. Decision-makers, not messengers.

**Cannot run without:** The pain-point inventory from Phase 2, the feasibility constraint brief, and a forced-ranking method (no "high/medium/high-medium" escape hatches).

**Must produce:**
- A single, ordered priority list with an explicit cut line
- Endorsement on the record from every stakeholder present
- Documented rationale for the top and bottom five (the contested ends)

---

## Solution Framing Workshop

**Purpose:** Translate prioritized problems into platform capabilities evaluated by effort and impact.

**Participants:** End users or their representatives, system owners, solution architects. 4–8.

**Cannot run without:** The prioritized problem list, feasibility dispositions, and someone empowered to say "that's out of scope" in the room.

**Must produce:**
- Defined problem statement per capability
- Agreed solution scope
- Prioritized feature concepts
- Draft acceptance criteria

**Facilitation note:** Hold the line on solutioning altitude. This event frames *what* the solution does, not *how* screens look. The room will want to design buttons; buttons are Phase 5's problem.

---

## Requirements Elaboration Session

**Purpose:** Decompose approved solutions into stories, acceptance criteria, and a delivery-ready backlog.

This is where framing becomes work a team can execute without a clarification loop. If you use a structured decomposition method, this is the session where it runs (see the companion [Requirements Engineering Framework](https://github.com/grahamdeno/requirements-engineering-framework)).

**Participants:** BA/product, tech leads, QA representation, user representatives on call for clarification.

**Cannot run without:** Framed solutions with draft acceptance criteria, the team's definition of ready, and style-reference examples of previously accepted stories.

**Must produce:**
- INVEST-compliant stories with testable acceptance criteria
- Traceability from every story back to a framed solution and a discovered need
- A flagged list of source gaps and open questions with owners. Write requirements at the altitude the source supports; filling gaps with optimism baselines a lie.
