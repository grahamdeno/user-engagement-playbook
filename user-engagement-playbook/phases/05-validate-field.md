# Phase 5 – Validate & Field

Six events across the build and fielding window. The sequence matters: iterate with users during the build, rehearse internally, test formally, and prepare the organization before go-live. Test events escalate in formality by design, from informal feedback to acceptance on the record.

---

## Prototype & Demo Feedback Session

**Purpose:** Put working software in front of users early and structure the feedback so it converts to backlog, not applause.

**Participants:** End users, system owners. 4–8.

**Cannot run without:** A working prototype or increment, defined feedback focus areas, and a capture method beyond memory.

**Must produce:**
- Annotated feedback log
- Identified usability issues
- Prioritized refinement backlog
- Updated / validated user stories

**Facilitation note:** A demo performed *at* users produces polite nods. Give users the keyboard. The difference in findings is the difference between watching someone drive and driving.

---

## Usability Testing

**Purpose:** Observe real users attempting real tasks on a stable build, with users as quality partners.

This event carries a second, internal mission: **training needs observation**. Where users hesitate, misread, or improvise is exactly the raw data that defines what training must exist. The observation happens here; the training needs analysis itself is an internal deliverable produced afterward, with no further user involvement. One session, two products.

**Participants:** End users executing scenarios; facilitator; dedicated observers.

**Cannot run without:** A stable prototype or MVP build, test scenarios and facilitator scripts, defined success metrics and task completion criteria, and data collection instruments (observation forms, usability survey).

**Must produce:**
- Usability test report
- Defect / issue log
- UX improvement recommendations
- Internal follow-on: training needs analysis memo derived from the observations

---

## Pre-UAT Testing

**Purpose:** Internal rehearsal with power users to validate the test scripts and catch blockers before formal acceptance.

The rehearsal. You do not discover during formal acceptance that the scenarios are broken, the test data is wrong, or defect #1 blocks scripts 2 through 30. Pre-UAT exists so UAT tests the *system*, not your preparation.

**Participants:** Power users and the internal test team. No formal acceptance authority present; that's the point. Rehearsals are where it's safe to fail.

**Must produce:** Validated (or corrected) test scripts, a cleared blocker list, and a go/no-go recommendation for entering UAT. **Timing:** late in the final build sprint, immediately before the UAT window.

---

## User Acceptance Testing (UAT)

**Purpose:** Formal acceptance testing by designated user representatives against defined acceptance criteria, with a post-test debrief before go-live.

Call it what it is. Naming this anything other than UAT obscures the one event whose name every stakeholder already understands. This is formal acceptance: criteria-based, on the record, and the moment ownership genuinely begins transferring to the user community.

**Participants:** Designated user representatives with acceptance authority, QA, BA/product, system owners.

**Must produce:**
- Pass/fail results against acceptance criteria, traceable to requirements
- Accepted-defect register with severity and disposition
- Formal acceptance decision, documented
- Post-UAT debrief: the retrospective that feeds the next release's process

---

## Change & Policy Readiness Session

**Purpose:** Identify policy updates, training requirements, and organizational actions needed before go-live.

The system can be perfect and the fielding still fails if the organization isn't prepared to receive it. This session inventories the non-software work: policy that must change, training that must exist (fed by the usability observations), communications that must go out, and the decisions only leadership can make.

**Participants:** Business owners, policy owners, training leads, comms, program leadership.

**Must produce:** A readiness action plan with owners and dates, keyed to the go-live date, and an escalation list of actions only leadership can unblock.

---

## Fielding Readiness Session

**Purpose:** Final verification that the system, the support structure, and the organization are prepared for go-live.

**Participants:** Program leadership, workstream owners, support/sustainment leads.

**Must produce:** A go/no-go readiness assessment against the readiness action plan, a day-one support plan, and a rollback trigger definition. **Timing:** early enough before go-live that readiness gaps can actually be closed; a readiness review held the day prior is a formality, not a review.
